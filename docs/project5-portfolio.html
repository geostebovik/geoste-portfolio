What I built

A production-pattern load balancer from scratch

This project deploys a complete high-availability web tier: two nginx VMs spread across separate fault domains in an availability set, fronted by a Standard SKU Azure Load Balancer with health probe monitoring and automatic traffic rerouting when a backend goes down.

The project took significantly longer than expected due to four distinct configuration issues — each of which exposed real Azure networking behaviour that documentation often glosses over. Those issues are documented in full below and on the Phase 2 lessons learned page.

Build sequence

How it came together

1
Availability set & VMs
Created avset-web with 2 fault domains and 2 update domains. Deployed vm-web-01 and vm-web-02 (Ubuntu 22.04, B1s) into vnet-spoke/snet-web from Phase 2 Project 3. Azure automatically placed each VM on a separate physical rack — if one rack loses power, the other keeps running.
2
Standard Load Balancer
Created lb-web (Standard SKU, public) with frontend IP pip-lb-web at 20.171.241.228. Added backend pool be-web, health probe probe-http (HTTP:80, 15s interval), and load balancing rule rule-http (TCP 80→80, no session persistence for round-robin).
3
Four-layer troubleshooting
The LB did not work immediately. Resolved in order: (1) browser HTTPS vs HTTP prefix, (2) missing AzureLoadBalancer NSG rule causing health probe failure, (3) missing outbound SNAT rule blocking nginx installation, (4) missing frontend IP config step before outbound rule creation. Each layer is documented with root cause and fix.
4
Nginx deployed & verified
Once outbound SNAT was configured, installed nginx on both VMs via Bastion. Customised the default page to display each VM's hostname. Verified round-robin distribution using a 20-request curl loop — both VMs received traffic confirming the LB was distributing correctly.
5
Failover test
Stopped nginx on vm-web-01. After two missed health probes (30 seconds), the portal showed vm-web-01 as Unhealthy and all traffic shifted to vm-web-02. Restarted nginx — vm-web-01 returned to the rotation within one probe cycle. Zero manual intervention required.

Lessons learned

Four issues that made this project genuinely educational

01
Browsers silently upgrade bare IPs to HTTPS
Opening 20.171.241.228 produced a "secure connections not supported" error. Chrome and Edge default to HTTPS for all addresses. Since the LB only had port 80 configured, the HTTPS attempt timed out. The fix is trivial — always type http:// explicitly. This makes a correctly configured LB appear broken and is worth knowing immediately when testing HTTP-only backends.
02
Standard LB health probes need an explicit NSG allow rule
Without a rule allowing the AzureLoadBalancer service tag on port 80, health probes are silently dropped by the NSG. Both VMs show as Down/Unhealthy and the LB drops all inbound traffic — producing a timeout that looks identical to the LB itself being misconfigured. Basic SKU allowed probe traffic implicitly. Standard SKU does not. This is a documented but easy-to-miss breaking change between SKUs.
03
Standard LB provides no implicit outbound SNAT for backend VMs
VMs in a Standard LB backend pool with no public IP have zero outbound internet access by default. Running sudo apt install nginx appeared to succeed but silently failed to download packages — the package manager printed connection warnings but reported success. The Unit nginx.service could not be found error was the first indication. Fix: create a second public IP, add it as a frontend IP config, then create an outbound rule. Always verify service installation with systemctl status.
04
The LB object model has a hidden intermediate layer the portal abstracts
Outbound rules reference frontend IP configurations, not public IPs directly. The dependency chain is: Public IP → Frontend IP Config → Rule. The CLI command failed with "not found" until the frontend IP config was created first. The portal also only shows frontend IP configs in dropdowns — not raw public IPs. Understanding this object model is necessary for CLI deployments and the AZ-104 exam.
Why this matters to employers

What this project signals beyond the configuration

Systematic troubleshooting under pressure
The load balancer worked after resolving four separate issues across the browser, NSG, outbound SNAT, and LB object model layers. Each was identified by reading error messages carefully, forming a hypothesis, and testing it — the same methodology used in production incident response. This project demonstrates that I don't just follow tutorials — I debug when things don't work.
Security-first NSG design using service tags
All NSG rules use Azure service tags (AzureLoadBalancer, VirtualNetwork, Internet) rather than hard-coded IP ranges. Service tags auto-update as Azure changes infrastructure IPs. The RDP deny rule correctly targets Internet rather than * so Bastion connections through the VNet are unaffected — a distinction that matters operationally.
Understanding Standard vs Basic SKU differences
Microsoft is retiring Basic SKU Load Balancer. Every new deployment should use Standard SKU — but Standard requires explicit configuration of things Basic provided implicitly (outbound SNAT, probe NSG rules, static public IPs). Organisations migrating existing deployments to Standard frequently hit exactly the issues documented here. Understanding the differences is immediately applicable in any Azure environment.
