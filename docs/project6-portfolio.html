What I built

Two DNS zones, one private endpoint, one locked-down storage account

This project covers two distinct capabilities that are often confused: public DNS zone management (creating authoritative records for a domain) and private endpoint networking (giving an Azure PaaS service a private IP address inside a VNet).

The end state: stlabeastus001.blob.core.windows.net resolves to a private IP address 10.0.1.6 when queried from inside the VNet, and returns a 404 WebContentNotFound when queried from the public internet. Same hostname. Different DNS answer depending on where the query originates. Zero application code changes required.

Core concept

A private endpoint creates a NIC with a private IP in your subnet and establishes an Azure Private Link connection to a PaaS service. The private DNS zone intercepts queries for the service hostname from inside linked VNets and returns that private IP instead of the public one. Traffic never leaves your VNet. The service doesn't need to know about it. This is how enterprises move PaaS services off the public internet without changing application configuration.

Access control outcome

Same hostname — completely different response depending on where you ask from

SourceDNS resolves toPublic accessResult
vm-web-01 (vnet-spoke) 10.0.1.6 (private) Disabled ✓ Access granted via private link
vm-hub-test (vnet-hub) 10.0.1.6 (private) Disabled ✓ Access granted via private link
Local browser (public internet) Public IP (overridden by Azure) Disabled ✗ 404 WebContentNotFound

The 404 WebContentNotFound response (rather than 403 Unauthorized) is deliberate. Azure Storage does not acknowledge the endpoint exists when public access is disabled — from the public internet, there is nothing to authenticate against. A 403 would reveal the resource exists. A 404 reveals nothing. This is a stronger security posture.

Why this matters to employers

What private endpoints signal about security maturity

Zero-trust network design for PaaS services
Private endpoints are the primary mechanism for removing PaaS services from the public internet without re-architecting applications. Any organisation subject to regulatory compliance (PCI DSS, HIPAA, ISO 27001) or enterprise security standards requires private endpoints for storage, SQL, Key Vault, and other PaaS services. Configuring and verifying this in a lab demonstrates readiness to implement it in regulated environments.
DNS as a security control — not just name resolution
This project demonstrates using DNS split-horizon to control traffic routing: same hostname resolves differently depending on network location. This is a sophisticated pattern used in enterprise environments to route internal traffic privately while keeping public-facing names unchanged. Understanding that DNS resolution and network routing are independent layers that must both be configured correctly is not an obvious insight.
Verified end-to-end — not just configured
The nslookup output showing a private IP, the 404 error from public access, and the continued private access from within the VNet together prove the entire private endpoint chain is functioning correctly: DNS zone linking, VNet peering carrying DNS queries across from spoke to hub, private endpoint NIC responding, and storage firewall enforcing the access restriction. Each component was individually verified.
