Bicep fundamentals — syntax, modules & what-if
Learn Infrastructure as Code by translating CLI commands you already know into declarative Bicep templates
⌄

Bicep
Key concept

Exam weight
20–25%
Cost

Analogy: CLI commands are like giving someone step-by-step cooking instructions. Bicep is like handing them a recipe card — you describe the finished dish and Azure figures out the steps. Declarative means you define the desired end state, not the sequence of operations to get there.
Why Bicep matters for the exam: The AZ-104 objectives (updated April 2026) explicitly require you to interpret Bicep files, modify existing templates, deploy resources using Bicep, and convert ARM JSON to Bicep. Expect 3–5 questions directly on Bicep syntax and deployment.
Steps
  • 1Install Bicep CLI: az bicep install. Install the Bicep extension in VS Code (publisher: Microsoft). Verify: az bicep version. This gives you IntelliSense, syntax highlighting, and inline error detection as you type.
  • 2Write your first Bicep file — a storage account. Create storage.bicep with a param for name, a variable for SKU, and a resource block. Deploy with: az deployment group create -g rg-az104-dev-wus3-01 -f storage.bicep. This is the equivalent of what you did with az storage account create.
  • 3Add a parameters file. Create dev.bicepparam — the new native Bicep parameter file format (replaces JSON parameter files). Reference it with --parameters dev.bicepparam. This is how you manage environment-specific values without touching the template.
  • 4Run what-if before every deployment: az deployment group what-if -g rg-az104-dev-wus3-01 -f storage.bicep. What-if shows exactly what will be created, modified, or deleted before anything happens. Build this as a mandatory habit — it is the IaC equivalent of reviewing a script before running it.
  • 5Extract a module. Move the storage account resource into modules/storage.bicep. Call it from a main.bicep using the module keyword. This is the lego pattern — reusable components assembled by a parent template.
  • 6Export an existing resource to Bicep. In the portal: go to any resource → Export template → Download. Then run az bicep decompile --file template.json to convert ARM JSON to Bicep. This is how you reverse-engineer existing infrastructure into IaC — a real-world skill for brownfield environments.
  • 7Translate your VM deploy script to Bicep. Take p9-quickdeploy-2vm.sh and rewrite it as Bicep. This is the highest-value exercise — you know what the output should be, so you can verify correctness. Use az bicep build to validate syntax without deploying.
Core Bicep syntax reference
bicep
// Parameters — inputs that change between deployments
param location string = 'westus3'
param environment string = 'dev'
param adminPassword @secure() string  // never logged or stored

// Variables — computed values reused within the template
var rgName = 'rg-az104-${environment}-wus3-01'
var tags = { env: environment, owner: 'ostebovik', project: 'az104' }

// Resource — declarative definition of an Azure resource
resource storageAccount 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: 'staz104dev${uniqueString(resourceGroup().id)}'
  location: location
  sku: { name: 'Standard_LRS' }
  kind: 'StorageV2'
  tags: tags
  properties: {
    allowBlobPublicAccess: false
    minimumTlsVersion: 'TLS1_2'
  }
}

// Module — call a child template
module network 'modules/networking.bicep' = {
  name: 'networkDeploy'
  params: {
    location: location
    environment: environment
  }
}

// Output — return values from the deployment
output storageAccountName string = storageAccount.name
output vnetId string = network.outputs.vnetId

Container Registry & Container Apps
Build, push and run containerized workloads — the production-correct alternative to Container Instances for web workloads
⌄
Registry
ACR Basic
Runtime
Container Apps
Key concept

Cost

Container Apps vs Container Instances — the exam distinction: ACI is a loading dock (bare, fast, disposable). Container Apps is a managed warehouse with auto-scaling, traffic splitting, and revision management. Both run containers — the workload pattern determines which to use. ACI = batch/testing. Container Apps = production web workloads.
⚠ Naming constraint: ACR names are alphanumeric only, no hyphens, globally unique. acraz104devwus301 — no separators between segments.
Steps
  • 1Create Azure Container Registry: az acr create -g rg-az104-dev-wus3-01 -n acraz104devwus301 --sku Basic. Basic SKU is sufficient for lab. Standard adds geo-replication, Premium adds private link support.
  • 2Build and push a container image using ACR Tasks — no local Docker required. Create a simple Dockerfile (nginx serving a custom page) and run: az acr build --registry acraz104devwus301 --image az104-web:v1 . Azure builds the image in the cloud and stores it in your registry.
  • 3Create a Container Apps environment: az containerapp env create -n cae-az104-dev-wus3-01 -g rg-az104-dev-wus3-01 -l westus3. The environment is the shared infrastructure that hosts one or more Container Apps — similar to an App Service Plan for App Service.
  • 4Deploy a Container App from ACR: az containerapp create pointing to your ACR image. Assign a system managed identity and grant it AcrPull role on the registry — no admin credentials needed.
  • 5Configure HTTP scaling: set min replicas to 0 (scales to zero when idle — no cost) and max to 3. This is the key cost advantage of Container Apps over App Service — you only pay when traffic exists.
  • 6Deploy a new revision: update the Dockerfile, rebuild with :v2 tag, create a new revision. Configure traffic splitting: 80% to v1, 20% to v2. This is the Container Apps equivalent of App Service deployment slots — canary deployments without downtime.
Key CLI commands
bash
# Build image in ACR (no local Docker needed)
az acr build --registry acraz104devwus301 \
  --image az104-web:v1 .

# Create Container Apps environment
az containerapp env create \
  -n cae-az104-dev-wus3-01 \
  -g rg-az104-dev-wus3-01 \
  -l westus3

# Deploy Container App with managed identity (no admin creds)
az containerapp create \
  -n ca-az104-dev-wus3-01 \
  -g rg-az104-dev-wus3-01 \
  --environment cae-az104-dev-wus3-01 \
  --image acraz104devwus301.azurecr.io/az104-web:v1 \
  --registry-server acraz104devwus301.azurecr.io \
  --min-replicas 0 --max-replicas 3 \
  --ingress external --target-port 80 \
  --assign-identity "[system]"

# Traffic split — 80/20 canary between revisions
az containerapp ingress traffic set \
  -n ca-az104-dev-wus3-01 \
  -g rg-az104-dev-wus3-01 \
  --revision-weight v1=80 v2=20

Advanced networking — Application Gateway, Bastion & Network Watcher
Layer-7 load balancing, secure VM access without public IPs, and network diagnostics
⌄

Cost

⚠ Cost warning: Application Gateway v2 costs ~$0.007/hr + data processing even at idle. Azure Bastion Standard costs ~$0.19/hr. Deploy these for the lab exercise then delete immediately after — do not leave running overnight.
App Gateway vs Load Balancer: The Phase 2 load balancer operates at Layer 4 (TCP/UDP) — it routes based on IP and port. Application Gateway operates at Layer 7 (HTTP/HTTPS) — it can route based on URL path, host header, cookies, and SSL certificates. Think of it as the difference between a traffic light (L4) and a traffic cop who reads your destination sign and directs you accordingly (L7).
Steps
  • 1Deploy Application Gateway v2 with WAF policy. Always use v2 (WAF_v2 or Standard_v2 SKU) — v1 is retired. Create a dedicated subnet snet-appgw-dev-wus3-01 with minimum /24. Application Gateway requires its own subnet — it cannot share with VMs or other resources.
  • 2Configure backend pool pointing to your App Service or VM. Configure HTTP settings (port 80 or 443), health probe (GET / every 30s), and listener. The health probe is how App Gateway determines if a backend is healthy before sending traffic.
  • 3Deploy Azure Bastion Standard SKU. Create snet-AzureBastionSubnet with minimum /26 — the subnet name must be exactly this. Standard SKU supports native RDP/SSH in browser and file transfer. Connect to vm-win-dev-01 via Bastion and verify BGInfo wallpaper.
  • 4Enable Network Watcher (auto-enabled per region — verify it exists): Monitor → Network Watcher. Run IP flow verify between your VMs to confirm NSG rules allow expected traffic. This is a diagnostic tool, not a resource you manage.
  • 5Create a Connection Monitor to test connectivity from vm-linux-dev-01 to vm-win-dev-01. Monitor the test for 5 minutes. Connection Monitor shows latency, packet loss, and topology — the operational view of your network.
  • 6Configure a user-defined route (UDR) to force traffic through a next-hop. Create a route table, add a route for 0.0.0.0/0 with next-hop type VirtualAppliance. Attach to a subnet. This is how you force traffic through Azure Firewall or an NVA in production hub-spoke topologies.

Key Vault & managed identity — zero-credential architecture
Store and retrieve secrets without credentials in code, scripts, or environment variables
⌄
Auth model
RBAC (not policies)
Access pattern
Managed identity
App integration
KV references
Cost
~$0 (10k ops free)
Analogy: Key Vault is a bank vault. Your managed identity is your ID badge — you show it to get in without carrying a key. The old approach (access policies) was like giving everyone a physical key to the vault and hoping they don't lose it. RBAC + managed identity means nobody carries a key at all.
⚠ Access Policies are deprecated for new vaults. Always create Key Vault with --enable-rbac-authorization true. If your guide says "Access Policies → Add", it is outdated. Use IAM → Add role assignment → Key Vault Secrets User instead.
Steps
  • 1Create Key Vault with RBAC authorization enabled (already done in bootstrap). Verify: portal → kv-az104-dev-wus3-01 → Access configuration → Permission model = Azure role-based access control.
  • 2Store secrets: az keyvault secret set --vault-name kv-az104-dev-wus3-01 --name db-connection-string --value "Server=...". Also store an admin password. These are the secrets your applications will need — never put these in code or environment variables directly.
  • 3Assign Key Vault Secrets User role to your App Service managed identity. This gives the App Service read-only access to secrets — it cannot create, modify, or delete. Least privilege applied correctly.
  • 4Configure Key Vault references in App Service: in App Settings, add a setting with value @Microsoft.KeyVault(SecretUri=https://kv-az104-dev-wus3-01.vault.azure.net/secrets/db-connection-string/). The App Service resolves this at runtime — the actual secret value never appears in the portal or in logs.
  • 5Verify the reference resolved: App Service → Configuration → the setting should show a green checkmark and "Key Vault Reference" status. A red X means the managed identity doesn't have access — check the role assignment.
  • 6Enable Key Vault diagnostic settings → send audit logs to your Log Analytics workspace. Every secret access is now logged — you can query who accessed what secret and when. This is security auditing at the data access level.
Key CLI commands
bash
# Store secrets (never hardcode these values in scripts you commit)

# Get App Service managed identity principal ID

  -g rg-az104-dev-wus3-01 \

  • 3Write data-tier.bicep. Deploy Azure SQL Server with Microsoft Entra admin (no SQL auth). Add the database with serverless tier to minimise cost. Configure private endpoint in the data subnet. Disable public network access.
  • 4Write api-tier.bicep. Deploy App Service Plan and API app with system managed identity. Configure VNet integration for outbound traffic. Create private endpoint so the web tier reaches the API internally. Add Key Vault reference for the SQL connection string.
  • 5Write web-tier.bicep. Deploy frontend App Service with system managed identity. Configure VNet integration. Add HTTPS-only and disable FTP. Create a staging slot. The frontend calls the API via its private endpoint — never directly to SQL.
  • 6Write monitoring.bicep. Add diagnostic settings on every resource pointing to your LAW. Create CPU alert on both App Service plans. Create failed request alert. Reuse the action group from Project 9.
  • 7Write main.bicep to orchestrate all modules in dependency order. Use Bicep's implicit dependency (symbolic names) rather than dependsOn wherever possible. Deploy the full stack: az deployment group create -g rg-az104-dev-wus3-01 -f main.bicep -p parameters/dev.bicepparam

This document is the most valuable portfolio artifact — it proves architectural judgment, not just execution.
Design decisions to document
decisions
Why App Service instead of VMs for web/api tiers?
  → Managed runtime, deployment slots, auto-scale
  → No OS patching, no agent management
  → Scale: would use AKS for microservices at enterprise scale

Why private endpoints on SQL and API?
  → Traffic never leaves Azure backbone
  → No public internet exposure for data tier
  → Scale: same pattern, add Azure Firewall in hub for east-west inspection

Why serverless SQL tier?
  → Pauses when idle — zero compute cost during lab off-hours
  → Scale: General Purpose or Business Critical for production SLA

Why Key Vault references instead of app settings?
  → Secrets never appear in portal, deployment logs, or git history
  → Centralised rotation — update KV secret, all apps get new value on restart
  → Scale: same pattern, add certificate management for TLS

Why system-assigned over user-assigned managed identity?
  → Lifecycle tied to resource — identity deleted with resource
  → Scale: user-assigned identity when multiple services share the same identity
           or when identity must exist before the resource (e.g. AKS node pools)
