AVOIDVNet peering is non-transitive. Hub-spoke requires peering from each spoke to hub — spokes cannot reach each other through the hub without NVA or Azure Firewall.
AVOIDNSG rule priorities are per-direction. Priority 130 inbound and 130 outbound do not conflict — Azure evaluates them in separate queues.
AVOIDUser delegation SAS fails for VM agent context — CSE uses service SAS (account key) only. VM agent has no Azure AD context.
EXAMStandard Load Balancer requires explicit outbound rules or NAT Gateway for outbound internet access. Basic SKU had implicit outbound — Standard does not.
EXAMPrivate endpoint DNS: requires private DNS zone linked to VNet. Without it, FQDN resolves to public IP even though traffic goes through private endpoint.
EXAMSplit-horizon DNS: same FQDN resolves to private IP inside VNet, public IP outside. Achieved with private DNS zone + VNet link.
KEY--nsg null detaches NSG from subnet. --nsg "" fails. --remove networkSecurityGroup also works.
KEYStatic private IPs on VMs: declare in Bicep AND set via CLI on existing NICs. CLI alone is overwritten on next Bicep deploy.
KEYStorage lifecycle policy tiers: Hot → Cool → Cold → Archive. Archive requires rehydrate before access (hours delay).
TIPHealth probe failure = LB stops sending traffic to that backend. Test by stopping IIS/nginx on one VM and verifying traffic shifts.
TIPSAS token scope: account SAS (all services), service SAS (one service), user delegation SAS (Entra ID backed — most secure for human access).

Virtual Networks, Subnets, NSGs & VNet Peering
Hub-spoke topology · bidirectional peering · NSG layered security · four subnets
⌄
Skill domain
Networking
Service model
IaaS
Est. cost
~$2
Status
Complete
Hub-spoke is the foundational Azure network pattern. The hub VNet hosts shared services (Bastion, firewall, DNS). Spoke VNets host workloads and peer back to the hub. Spokes cannot communicate directly — all inter-spoke traffic routes through the hub. NSGs at the subnet level enforce traffic rules independent of VNet peering.
What was built
  • 1
    Hub VNet vnet-hub-dev-wus3-01 with four subnets: web, api, data, bastion
  • 2
    Spoke VNet vnet-spoke-dev-wus3-01 peered bidirectionally to hub
  • 3
    NSG per subnet with layered inbound/outbound rules — deny-all default with explicit allows
  • 4
    Azure Bastion in hub for secure VM access without public IPs
  • 5
    Verified cross-VNet connectivity and NSG blocking behavior
Key CLI commands
bash 
# Create hub VNet
az network vnet create -g $RG -n vnet-hub-dev-wus3-01 \
  --address-prefix 10.0.0.0/16 -l $LOC

# Create subnet with NSG attached
az network vnet subnet create -g $RG \
  --vnet-name vnet-hub-dev-wus3-01 \
  -n snet-web-dev-wus3-01 \
  --address-prefix 10.0.1.0/24 \
  --network-security-group nsg-web-dev-wus3-01

# Peer hub to spoke (bidirectional — run for both directions)
az network vnet peering create -g $RG \
  -n peer-hub-to-spoke \
  --vnet-name vnet-hub-dev-wus3-01 \
  --remote-vnet vnet-spoke-dev-wus3-01 \
  --allow-vnet-access

Storage Accounts — Blobs, SAS, Lifecycle & Static Website
LRS vs GRS · SAS tokens · lifecycle tiering · soft delete · key rotation · static portfolio page

Storage accounts are the foundation of Azure data services. Redundancy (LRS/GRS/ZRS) determines availability guarantees. SAS tokens scope access without exposing account keys. Lifecycle policies automate cost management by tiering blobs from Hot to Cool to Archive based on last-modified date.
What was built
  • 1
    LRS storage account for cost-optimized lab storage — stdevwus301
  • 2
    Blob container with public access disabled — SAS token required for access
  • 3
    Service SAS token scoped to blob read — time-limited, no account key exposure
  • 4
    Lifecycle policy: Hot → Cool after 30 days, Cool → Archive after 90 days
  • 5
    Soft delete enabled — 7-day recovery window
  • 6
    Static website hosting enabled — $web container serves index.html
⚠ SAS token scope matters: user delegation SAS (Entra ID backed) is most secure for human access. Service SAS (account key) required for VM agent context (CSE). Never use account keys directly in application code.
Key CLI commands
bash 
# Generate service SAS for blob read
az storage blob generate-sas \
  --account-name stdevwus301 \
  --container-name content \
  --name file.txt \
  --permissions r \
  --expiry 2026-01-01 \
  --auth-mode login -o tsv

# Enable static website hosting
az storage blob service-properties update \
  --account-name stdevwus301 \
  --static-website \
  --index-document index.html \
  --404-document 404.html

Azure Load Balancer & High Availability
Standard SKU · availability set · health probes · failover testing · outbound SNAT
⌄
Skill domain
Networking
Service model
IaaS
Est. cost
~$5
Status
Complete
Azure Load Balancer operates at Layer 4 (TCP/UDP). Standard SKU is zone-redundant and required for production. Health probes determine which backend instances receive traffic — a failed probe removes the instance from rotation automatically. Standard SKU requires explicit outbound rules; unlike Basic SKU there is no implicit outbound internet access.
What was built
  • 1
    Standard SKU public load balancer lb-web-dev-wus3-01 with frontend IP
  • 2
    Backend pool with two VMs in availability set — ensures VMs are on separate fault domains
  • 3
    HTTP health probe on port 80 — 15s interval, 2 unhealthy threshold
  • 4
    Load balancing rule: frontend port 80 → backend port 80
  • 5
    Outbound rule for SNAT — explicit outbound internet access for backend VMs
  • 6
    Failover test: stopped IIS on VM1, verified all traffic routed to VM2
⚠ Load Balancer vs Application Gateway: LB is Layer 4 (IP/port routing only). App Gateway is Layer 7 (HTTP/HTTPS, path-based routing, WAF). Use LB for non-HTTP workloads or when you need the lowest latency. Use App Gateway when you need SSL termination, URL routing, or WAF.

Azure DNS & Private Endpoints
Public + private DNS zones · private endpoint for blob · split-horizon resolution · public access lockdown
⌄
Skill domain
Networking
Service model
IaaS · PaaS
Est. cost
~$4
Status

Private endpoints give PaaS services (Storage, SQL, Key Vault) a private IP inside your VNet. Traffic never leaves the Microsoft backbone. Split-horizon DNS means the same FQDN resolves to a private IP inside the VNet and a public IP outside — achieved with a private DNS zone linked to the VNet and Azure DNS resolver integration.
What was built
  • 1
    Public DNS zone ostebovik.net delegated to Azure DNS
  • 2
    Private endpoint for stdevwus301 blob service — private IP in VNet
  • 3
    Private DNS zone privatelink.blob.core.windows.net linked to VNet
  • 4
    Public blob access disabled — storage firewall set to deny all
  • 5
    Verified: nslookup from inside VNet resolves to private IP, from outside resolves to public IP
Key CLI commands
bash 
# Create private endpoint for storage blob
az network private-endpoint create -g $RG \
  -n pep-st-dev-wus3-01 \
  --vnet-name vnet-hub-dev-wus3-01 \
  --subnet snet-data-dev-wus3-01 \
  --private-connection-resource-id $ST_ID \
  --group-id blob \
  --connection-name conn-st-blob

# Link private DNS zone to VNet
az network private-dns link vnet create -g $RG \
  -n link-blob-vnet \
  --zone-name privatelink.blob.core.windows.net \
  --virtual-network vnet-hub-dev-wus3-01 \
  --registration-enabled false
