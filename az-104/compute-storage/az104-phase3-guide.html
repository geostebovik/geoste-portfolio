Virtual machines — deployment, extensions & disk management
Simulate enterprise VM provisioning with custom scripts, availability, and managed disk operations
⌄
VMs deployed
2 (Win + Linux)
Extensions
Custom Script + BGInfo
Disk ops
Attach, resize, snapshot
Availability
Availability zones
Analogy: A VM is like renting a physical server but Azure manages the hardware underneath. The availability zone is like choosing which building your server lives in — if one building loses power, the others keep running. VM Extensions are like app installers that run automatically at first boot without you ever logging in.
Portal steps
  • 1Create resource group rg-phase3-compute in East US. This is your Phase 3 sandbox — all compute resources live here.
  • 2Deploy a Windows Server VM: Create VM → name vm-win-01, image Windows Server 2022 Datacenter, size Standard_B2s, Zone 1. No public IP. Use vnet-spoke / snet-web from Phase 2. Generate SSH keypair and save it.
  • 3Deploy a Linux VM: name vm-linux-01, Ubuntu 22.04, Standard_B1s, Zone 2. Same VNet, no public IP. This cross-zone pair demonstrates zone-redundant architecture.
  • 4Add a Custom Script Extension to vm-win-01: VM → Extensions + Applications → Add → Custom Script Extension. Upload a .ps1 script that installs IIS and creates a test page. Azure executes this at extension install time — no RDP needed. This is how enterprises automate post-deployment configuration.
  • 5Add BGInfo extension to vm-win-01: Extensions → Add → BGInfo. This writes system info to the desktop wallpaper — useful for confirming which VM you're connected to. Screenshot it via Bastion after install.
  • 6Attach a data disk to vm-linux-01: VM → Disks → Add data disk. Create a new 32GB Premium SSD managed disk named disk-data-linux01. Save. Then connect via Bastion and initialize it: sudo fdisk -l to find it, sudo mkfs.ext4 /dev/sdc to format, sudo mount /dev/sdc /mnt/data to mount.
  • 7Resize the OS disk on vm-linux-01: deallocate the VM first (this is required — you cannot resize a disk on a running VM). In Disks → click the OS disk → Size + performance → change from 30GB to 64GB. Restart the VM. Inside the VM run sudo growpart /dev/sda 1 && sudo resize2fs /dev/sda1 to expand the partition.
  • 8Create a disk snapshot: Disks → select disk-data-linux01 → Create snapshot. Name: snap-data-linux01-001. This is point-in-time backup of a managed disk — faster than a full VM backup for data disks.
  • 9Review VM metrics in the portal: VM → Monitoring → Metrics. Plot CPU Percentage and Disk Read Bytes over the last hour. This is a preview of Project 9 — notice Azure collects these automatically with no configuration needed.
  • 10Run the Azure VM Run Command: VM → Run command → RunPowerShellScript (Windows) or RunShellScript (Linux). Enter hostname and click Run. Output appears in the portal. This lets you run commands on a VM without Bastion or SSH — critical for emergency access when networking is broken.
Key CLI commands
bash 
# Deploy Linux VM in zone 2, no public IP
az vm create -g rg-phase3-compute -n vm-linux-01 \
  --image Ubuntu2204 --size Standard_B1s --zone 2 \
  --vnet-name vnet-spoke --subnet snet-web \
  --admin-username azureuser --generate-ssh-keys \
  --public-ip-address "" --nsg ""

# Add and attach a 32GB data disk
az vm disk attach -g rg-phase3-compute --vm-name vm-linux-01 \
  --name disk-data-linux01 --size-gb 32 \
  --sku Premium_LRS --new

# Snapshot the data disk
DISK_ID=$(az disk show -g rg-phase3-compute \
  -n disk-data-linux01 --query id -o tsv)
az snapshot create -g rg-phase3-compute \
  -n snap-data-linux01-001 --source $DISK_ID

# Run a command on VM without SSH
az vm run-command invoke -g rg-phase3-compute \
  -n vm-linux-01 --command-id RunShellScript \
  --scripts "hostname && df -h && free -m"

# Deallocate VM before disk resize
az vm deallocate -g rg-phase3-compute -n vm-linux-01
az disk update -g rg-phase3-compute \
  -n disk-data-linux01 --size-gb 64
az vm start -g rg-phase3-compute -n vm-linux-01

App Service, deployment slots & Container Instances
Platform-managed web hosting with zero-downtime deployments and lightweight container workloads
⌄
App Service plan
Standard S1
Web apps
1 with 2 slots
Deployment
Slot swap (blue/green)
Container
Azure Container Instance
Analogy: App Service is like a managed apartment building — Azure handles plumbing, electricity, and structure; you just move in your app. A deployment slot is like having a second apartment in the same building — you test changes there, then do a swap (swap the front doors) with zero downtime. No one notices the move happened.
Portal steps
  • 1Create an App Service Plan: search "App Service plans" → Create. Name: asp-phase3, OS: Linux, region: East US, SKU: Standard S1. Standard tier is required for deployment slots — Free and Basic don't support them. Cost ~$0.10/hr so delete promptly.
  • 2Create a Web App: App Services → Create. Name: app-az104-yourname (must be globally unique — it becomes your URL). Runtime: Python 3.11 or Node 18. Plan: asp-phase3. Deploy a simple hello-world app via the portal's built-in editor or GitHub integration.
  • 3Add a deployment slot: Web App → Deployment slots → Add slot. Name: staging. Clone settings from production. You now have two separate environments at app-az104-yourname.azurewebsites.net (production) and app-az104-yourname-staging.azurewebsites.net (staging).
  • 4Deploy a different version to staging: edit the index page in staging to show "VERSION 2 — STAGING". Verify it at the staging URL. Production still shows the original content.
  • 5Perform a slot swap: Deployment slots → Swap. Source: staging, Target: production. Click Swap. Within 30 seconds, production now serves VERSION 2 and staging serves the old version. No downtime occurred. Screenshot the swap completion. This is blue/green deployment — a critical DevOps skill.
  • 6Configure App Settings: Web App → Configuration → Application settings → New. Add ENVIRONMENT=production and APP_VERSION=2.0. These become environment variables in the app. Mark ENVIRONMENT as a "Slot setting" so it stays with the slot during swaps (doesn't travel with the code).
  • 7Enable Always On: Configuration → General settings → Always On: On. Without this, the app spins down after 20 minutes of inactivity and the first request takes 10–30 seconds. Required for production workloads.
  • 8Deploy a Container Instance: search "Container Instances" → Create. Name: aci-nginx-lab, image: nginx:latest (public Docker Hub), OS: Linux, CPU: 1, Memory: 1.5GB. Set DNS label: aci-nginx-yourname. Within 60 seconds you have a running nginx at a public URL — no VM, no App Service plan. This is the lightest-weight compute option in Azure.
  • 9Check container logs: Container Instance → Containers → Logs. You can see nginx access logs in real time in the portal. Also try Connect tab — it gives you a shell inside the running container. Run nginx -v to confirm the version.
Key CLI commands
bash 
# Create App Service plan (Standard tier for slots)
az appservice plan create -g rg-phase3-compute \
  -n asp-phase3 --sku S1 --is-linux

# Create web app
az webapp create -g rg-phase3-compute \
  -p asp-phase3 -n app-az104-yourname \
  --runtime "PYTHON:3.11"

# Add staging deployment slot
az webapp deployment slot create -g rg-phase3-compute \
  -n app-az104-yourname --slot staging \
  --configuration-source app-az104-yourname

# Add app settings (slot-sticky)
az webapp config appsettings set -g rg-phase3-compute \
  -n app-az104-yourname \
  --settings ENVIRONMENT=production APP_VERSION=2.0
az webapp config appsettings set -g rg-phase3-compute \
  -n app-az104-yourname --slot-settings ENVIRONMENT=production

# Swap staging → production
az webapp deployment slot swap -g rg-phase3-compute \
  -n app-az104-yourname \
  --slot staging --target-slot production

# Deploy nginx container instance
az container create -g rg-phase3-compute \
  -n aci-nginx-lab \
  --image nginx:latest --ports 80 \
  --cpu 1 --memory 1.5 \
  --dns-name-label aci-nginx-yourname \
  --os-type Linux

# Get container public URL
az container show -g rg-phase3-compute \
  -n aci-nginx-lab \
  --query "ipAddress.fqdn" -o tsv

Azure Monitor — Log Analytics, alerts & workbooks
Build the observability layer that tells you when things go wrong before users notice
⌄
Log Analytics
1 workspace
Alert rules
3 (metric + log)
Action group
Email notifications
Workbook
Custom dashboard
Analogy: Azure Monitor is like the building's central security and facilities system. Log Analytics is the CCTV recording everything to a searchable archive. Metric alerts are motion sensors — they fire when a reading crosses a threshold. Log alerts are pattern matchers — they search recordings for specific events. Action Groups are the alarm contacts list — who gets called and how.
Portal steps
  • 1Create a Log Analytics workspace: search "Log Analytics workspaces" → Create. Name: law-phase3-monitor, resource group: rg-phase3-compute, region: East US. This is the central repository for all log data across your resources.
  • 2Connect VMs to the workspace: go to vm-linux-01 → Monitoring → Diagnostic settings → Add diagnostic setting. Send all categories to law-phase3-monitor. Repeat for vm-win-01. Also install the Azure Monitor Agent: VM → Extensions → Add → Azure Monitor Agent.
  • 3Create an Action Group first (needed before alert rules): Monitor → Action groups → Create. Name: ag-phase3-ops. Add action: Email/SMS/Push/Voice. Enter your email address. Action name: email-me. Save. This is the "who gets notified" list.
  • 4Create a CPU metric alert: Monitor → Alerts → Create → Alert rule. Scope: vm-linux-01. Signal: Percentage CPU. Condition: greater than 80%, evaluation every 1 minute, lookback 5 minutes. Severity: 2 (Warning). Action group: ag-phase3-ops. Name: alert-cpu-high-linux01.
  • 5Trigger the CPU alert to test it: connect to vm-linux-01 via Bastion and run stress --cpu 4 --timeout 300 (install first: sudo apt install stress -y). Within 5 minutes you should receive an email alert. Screenshot the fired alert in Monitor → Alerts.
  • 6Create a log-based alert: Monitor → Alerts → Create. Scope: law-phase3-monitor. Signal type: Custom log search. Query: Heartbeat | where TimeGenerated > ago(5m) | summarize count() by Computer | where count_ < 1. This fires if a VM stops sending heartbeats — i.e., it's down. Threshold: greater than 0 results. This is a VM availability alert.
  • 7Explore Log Analytics queries: law-phase3-monitor → Logs. Run these KQL queries and screenshot the results:
    — Heartbeat | summarize LastHeartbeat=max(TimeGenerated) by Computer
    — Perf | where CounterName == "% Processor Time" | summarize avg(CounterValue) by Computer, bin(TimeGenerated, 5m)
  • 8Create a Workbook: Monitor → Workbooks → New. Add a metric chart showing CPU across both VMs. Add a log query tile showing last 24h of heartbeats. Add a text tile explaining what the dashboard shows. Save as workbook-phase3-vms. Pin it to your Azure dashboard. This is the kind of operational dashboard you'd hand to an L1 support team.
  • 9Set up diagnostic settings on the storage account from Phase 2: stlabeastus001 → Diagnostic settings → Add. Send StorageRead, StorageWrite, StorageDelete logs to law-phase3-monitor. This lets you query who accessed what blob and when — essential for security auditing.
Key KQL queries + CLI
bash / KQL 
# Create Log Analytics workspace
az monitor log-analytics workspace create \
  -g rg-phase3-compute \
  -n law-phase3-monitor \
  --location eastus \
  --sku PerGB2018

# Create action group with email
az monitor action-group create \
  -g rg-phase3-compute \
  -n ag-phase3-ops \
  --short-name phase3ops \
  --action email email-me your@email.com

# Create CPU metric alert
VM_ID=$(az vm show -g rg-phase3-compute \
  -n vm-linux-01 --query id -o tsv)
az monitor metrics alert create \
  -g rg-phase3-compute \
  -n alert-cpu-high-linux01 \
  --scopes $VM_ID \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 2 \
  --action ag-phase3-ops

--- KQL: Last heartbeat per VM ---
Heartbeat
| summarize LastSeen=max(TimeGenerated) by Computer
| extend MinutesSinceLastSeen=datetime_diff('minute',now(),LastSeen)
| order by MinutesSinceLastSeen asc

--- KQL: Average CPU over time ---
Perf
| where CounterName == "% Processor Time"
| where ObjectName == "Processor"
| where InstanceName == "_Total"
| summarize AvgCPU=avg(CounterValue)
    by Computer, bin(TimeGenerated, 5m)
| render timechart

Azure Backup & Site Recovery fundamentals
Protect workloads with recovery vaults, backup policies, and DR replication concepts
⌄
Recovery vault
1 created
Backup policy
Custom schedule
VM backup
Triggered + verified
File recovery
Item-level restore
Analogy: The Recovery Services vault is like a fireproof safe in a separate building from your office. Backup policy is the schedule for how often someone carries documents to that safe. Azure Site Recovery is like having a full duplicate office ready in another city — if your main office burns down, staff show up at the duplicate and work resumes within hours instead of weeks.
Portal steps
  • 1Create a Recovery Services vault: search "Recovery Services vaults" → Create. Name: rsv-phase3-backup, resource group: rg-phase3-compute, region: East US. Important: the vault must be in the same region as the VMs you want to back up.
  • 2Configure vault redundancy: rsv-phase3-backup → Properties → Backup configuration → Update. Set Storage replication type to Locally Redundant (LRS). For a lab, LRS is fine and cheaper. Production vaults typically use GRS so backups survive a regional outage.
  • 3Create a custom backup policy: vault → Backup policies → Add. Policy type: Azure Virtual Machine. Name: pol-daily-7day. Schedule: Daily at 11:00 PM UTC. Retention: daily 7 days, weekly 4 weeks, monthly 3 months. This is a realistic production policy — daily backups with tiered retention.
  • 4Enable backup for vm-linux-01: vault → Backup → Azure Virtual Machine → Add. Select vm-linux-01, assign policy pol-daily-7day. Alternatively: VM → Backup → Enable backup (points to same flow). After enabling, click "Backup now" to trigger an immediate backup rather than waiting for the scheduled time.
  • 5Monitor the backup job: vault → Backup Jobs. Watch the job progress through Triggered → Taking Snapshot → Transferring data to vault → Completed. This takes 10–20 minutes for a B1s VM. Screenshot the Completed status — this is your proof of working backup.
  • 6Explore restore options: vault → Backup Items → Azure Virtual Machine → vm-linux-01 → Restore VM. Note the three options: Create new VM (deploys a new VM from backup), Replace existing (overwrites the source VM), Restore disks (gives you managed disks to attach anywhere). Do NOT click Restore in the lab — just screenshot the options. Understanding restore paths is exam content.
  • 7File-level recovery: vault → Backup Items → vm-linux-01 → File Recovery. Select a recovery point. Download the script provided — this mounts the backup as a network share on any VM so you can browse and copy individual files without restoring the entire VM. This is how you recover a single accidentally-deleted config file in production.
  • 8Enable soft delete on the vault: vault → Properties → Security settings → Soft delete: Enabled. With soft delete, even if a backup is deleted, it's retained for 14 days and can be recovered. This protects against ransomware attacks that try to destroy backups before encrypting VMs.
  • 9Azure Site Recovery concepts: vault → Site Recovery → Step 1: Replicate. Walk through the wizard without completing it — observe the settings: source region (East US), target region (West US 2), replication policy (RPO 24h, recovery point retention 72h). Screenshot the configuration wizard. ASR is too expensive to run in a lab ($25+/month per VM) so documenting the configuration options is the right approach here.
Key CLI commands
bash 
# Create Recovery Services vault
az backup vault create \
  -g rg-phase3-compute \
  -n rsv-phase3-backup \
  --location eastus

# Set vault storage redundancy to LRS
az backup vault backup-properties set \
  -g rg-phase3-compute \
  -n rsv-phase3-backup \
  --backup-storage-redundancy LocallyRedundant

# Enable VM backup with default policy first
VM_ID=$(az vm show -g rg-phase3-compute \
  -n vm-linux-01 --query id -o tsv)
az backup protection enable-for-vm \
  -g rg-phase3-compute \
  -v rsv-phase3-backup \
  --vm $VM_ID \
  --policy-name DefaultPolicy

# Trigger an immediate backup now
az backup protection backup-now \
  -g rg-phase3-compute \
  -v rsv-phase3-backup \
  --container-name "IaasVMContainer;iaasvmcontainerv2;rg-phase3-compute;vm-linux-01" \
  --item-name "vm;iaasvmcontainerv2;rg-phase3-compute;vm-linux-01" \
  --retain-until $(date -u -d "+7 days" '+%d-%m-%Y')

# Monitor backup jobs
az backup job list \
  -g rg-phase3-compute \
  -v rsv-phase3-backup \
  --output table

# List restore points once backup completes
az backup recoverypoint list \
  -g rg-phase3-compute \
  -v rsv-phase3-backup \
  --container-name "IaasVMContainer;iaasvmcontainerv2;rg-phase3-compute;vm-linux-01" \
  --item-name "vm;iaasvmcontainerv2;rg-phase3-compute;vm-linux-01" \
  --output table
