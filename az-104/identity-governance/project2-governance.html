What I built

A baseline governance framework from scratch

In any real Azure environment, you can't wait until resources are deployed to think about compliance. Governance has to be in place first — before a single VM or storage account exists — otherwise you spend months cleaning up what should never have been allowed. This project builds exactly that foundation.

I created three Azure Policy assignments, bundled them into a reusable Policy Initiative, applied two types of resource locks, and configured a cost budget with automated alerts — all scoped to a dedicated lab resource group to prove the controls work without touching anything outside it.

Policies deployed

Three controls, three different enforcement modes

Azure Policy has several effects — Deny blocks non-compliant deployments outright, Audit flags them without blocking (useful for discovery), and Append automatically adds required fields. I used all three to show how a layered compliance approach works.

Code artifact

Policy initiative definition — exported from Azure

After testing, I exported the initiative definition as JSON. This is the artifact that can be imported into any Azure subscription to instantly apply the same governance baseline — this is how real organizations deploy policy at scale across hundreds of subscriptions using Azure DevOps or GitHub Actions.

JSON initiative-baseline-governance.json 
{
  "name": "initiative-baseline-governance",
  "properties": {
    "displayName": "Baseline Governance Initiative",
    "description": "Enforces location, tagging, and disk compliance",
    "policyDefinitions": [
      {
        // Deny deployments outside East US
        "policyDefinitionId": "/providers/Microsoft.Authorization/
          policyDefinitions/e56962a6-4747-49cd-b67b-bf8b01975c4f",
        "parameters": {
          "listOfAllowedLocations": {
            "value": ["westus3"]
          }
        }
      },
      {
        // Require 'environment' tag on resource groups
        "policyDefinitionId": "/providers/Microsoft.Authorization/
          policyDefinitions/96670d01-0a4d-4649-9c89-2d3abc0a5025",

Why this matters to employers

What this project demonstrates beyond the exam

Governance is the difference between a cloud environment that stays manageable at 10 subscriptions and one that collapses into chaos at 3. Every organization using Azure at scale has a governance story — and the Azure administrator is usually the one writing it. Here's what this project signals to a hiring team.

Security posture — not just configuration

Using Deny effect to block non-approved regions prevents data sovereignty violations before they happen. Using Audit before switching to Deny shows measured, production-safe rollout thinking — the same approach used in enterprise change management.

Compliance automation at scale

A Policy Initiative exported as JSON can be applied to 500 subscriptions in a single pipeline run. This project demonstrates understanding of that pattern — governance as code, not governance as portal clicking. That's the skill that matters in organizations managing cloud at scale.

Cost awareness built in from day one

Configuring a budget alert before doing any other work — not as an afterthought — signals FinOps awareness. Azure admins who understand that governance includes cost governance are significantly more valuable than those who treat billing as someone else's problem.

Operational discipline — testing what you build

Deliberately triggering the location policy denial and the resource lock error — and capturing both as evidence — shows that I don't just configure controls, I verify them. In a production environment, an untested policy is an unknown policy.

Lessons learned

What surprised me, and what I'd do differently

01
Policy evaluation is not immediate After assigning a policy, Azure takes 5–15 minutes to evaluate existing resources. I initially thought the policy wasn't working — it was just still propagating. In production, you'd use the compliance scan trigger to force immediate evaluation.
02
ReadOnly locks block more than expected A ReadOnly lock prevents not just writes to resource properties but also operations that require write access to metadata — including some read-only-seeming operations like listing storage keys. Understanding lock scope is critical before applying locks in production.
03
Initiatives are better than standalone assignments from day one Even with only three policies, grouping them into an initiative immediately made compliance reporting cleaner and assignment to new scopes a single action. In hindsight I'd always start with an initiative, never standalone assignments.
04
Forecast alerts are more useful than actual spend alerts A 100% actual spend alert tells you you've already overspent. A 100% forecast alert gives you time to react. In this lab I set both, but in a real environment the forecast alert is the one that matters.
